# 周视图轻松安排周计划设计

## 背景

PlanTodo 已完成日历排期一期和二期：任务支持未安排状态，日历页有周视图、月视图、列表视图，安排任务栏可以展示未安排任务并拖入日历。现在的短板不是排期模型，而是周视图里的操作路径仍偏重：创建时间段任务、创建跨天全天任务、调整任务时长和控制时间轴密度都不够顺手。

本设计参考滴答清单“周视图：轻松安排周计划”，但不复刻其全部日历能力。本期只解决周计划排期效率，标签、优先级、隐藏时段、多日视图、多周视图和周复盘导出不进入本规格。

## 已确认范围

本期采用小步增强现有周视图的方案，不重建日历引擎。

进入本期：

1. 周视图时间轴空白处快速创建时间段任务。
2. 周视图顶部全天栏拖选多个日期，创建跨天全天任务。
3. 两类快速创建都使用同一个原地浮层输入标题和分类。
4. 已有时间段任务支持拖动底部边缘调整时长。
5. 周视图时间轴支持紧凑、标准、宽松三档密度。
6. 完整日历页的安排任务栏提供显式入口、关闭能力和本页展开状态，继续承接未安排任务拖拽排期。

不进入本期：

- 月视图日期格拖选跨天任务。
- 隐藏非工作时段。
- 全天栏高度手动调整。
- 日视图、三日视图、多日视图、多周视图。
- 周复盘导出或摘要能力。
- 标签和优先级模型。
- 滴答式 `清单 / 标签 / 优先级` 完整侧栏。

标签和优先级是独立任务元数据模型，必须另起规格。把它们塞进本期会把周视图功能扩成任务系统重构。

## 产品行为

### 时间轴快速创建时间段任务

用户在周视图时间轴空白区域点击时，系统创建一个默认 60 分钟的待确认时间段。例如点击 `2026-06-18 09:00`，浮层显示 `09:00-10:00`。

用户在时间轴空白区域按下并纵向拖动时，系统按 15 分钟步进计算开始和结束时间。松开后显示原地浮层。浮层提交前不写数据库，取消不会产生占位任务。

保存时创建时间段任务：

```ts
{
  title,
  categoryId,
  plannedDate: 'YYYY-MM-DD',
  startAt: 'YYYY-MM-DDTHH:mm:00.000',
  endAt: 'YYYY-MM-DDTHH:mm:00.000',
  allDay: false
}
```

时间段任务不能跨天。开始时间最晚允许到 `23:44`，这样默认 15 分钟结束时间可以落在 `23:59`。点击或拖拽落点晚于 `23:44` 时，前端按 `23:44` 处理，而不是创建小于 15 分钟的任务，也不使用 `24:00`。

时间轴纵向拖拽支持向下和向上两个方向。无论从哪个方向拖，系统都取较早时间作为 `startAt`，较晚时间作为 `endAt`，并按 15 分钟步进吸附。吸附后不足 15 分钟时按 15 分钟处理；如果范围越过当天末尾，则先把结束夹到 `23:59`，再保证开始不晚于 `23:44`。

### 全天栏拖选创建跨天全天任务

用户在周视图顶部全天栏某一天按下并横向拖到另一日期，系统选中连续日期范围。松开后显示同一个原地浮层，标题区域上方展示日期范围，例如 `12月18日 - 12月21日`。

保存时创建跨天全天任务：

```ts
{
  title,
  categoryId,
  plannedDate: '2026-06-18',
  plannedEndDate: '2026-06-21',
  allDay: true
}
```

如果只选中一天，则创建单日全天任务，不写 `plannedEndDate`。

### 原地浮层

浮层用于两类创建场景：

- 时间段任务：显示时间范围。
- 全天/跨天任务：显示日期范围。

浮层字段：

- 任务标题，必填，trim 后不能为空。
- 分类，默认使用当前第一个分类；如果没有分类，显示错误并禁止保存。

浮层操作：

- Enter 保存。
- Esc 取消。
- 点击取消按钮或外部空白取消。
- 保存成功后刷新日历数据和安排任务栏。
- 保存失败时显示错误，浮层保留，用户输入不丢失。

### 调整时间段任务时长

已有时间段任务块底部显示可拖拽边缘。用户拖动底部边缘时，系统按 15 分钟步进调整结束时间。

规则：

- 最小时长 15 分钟。
- 不允许结束时间早于开始时间。
- 不允许拖出当前日期；超过当天范围时夹到 `23:59`。
- 如果已有任务开始时间晚于 `23:44`，尾部 resize 无法同时满足 15 分钟最小时长和不跨天约束。此类任务不显示 resize handle，用户需要先移动任务到更早时间再调整时长。
- 只调整 `endAt`，不改变 `plannedDate`、`startAt`、`categoryId`、`title`。
- 提交失败时显示错误并回到服务端数据状态。

本期只调整任务尾部，不做顶部拖拽修改开始时间。

### 时间轴密度

周视图提供三档时间轴密度：

| 密度 | 小时高度 | 用途 |
| --- | --- | --- |
| 紧凑 | 48px | 一屏查看更多时间 |
| 标准 | 64px | 默认视图 |
| 宽松 | 88px | 任务块更易读、更易拖拽 |

密度只影响周视图时间轴的小时行高度、任务块定位和拖拽换算，不影响后端数据。

状态并入现有 `CalendarSettings` 本地设置，不使用第二个 localStorage key。切换月视图或列表视图时不显示该控制；回到周视图时保留上次选择。

### 安排任务栏

当前安排任务栏已能加载未安排任务和当前范围内全天无时间任务。本期补齐产品入口和本页展开状态：

- 日历工具栏提供“安排任务”入口。
- 右侧栏可关闭。
- 关闭后日历主区域占满宽度。
- 打开后恢复右侧安排任务栏。
- 安排任务栏关闭状态只保存在当前页面组件状态中，不写入 localStorage。
- 现有搜索、分类过滤、多选、批量安排、批量取消安排和拖拽排期作为回归约束继续保持。
- 单任务可以拖到日期格、全天栏或时间轴；多选任务只能通过批量按钮或拖到日期/全天区域安排为全天任务，多选拖到时间轴继续被拒绝。

本期不加入标签和优先级 tab。侧栏可以按现有分类和排期状态组织内容，但不能展示没有真实模型支撑的假 tab。

## 架构设计

继续保持现有 calendar 模块边界：

- `CalendarPanel` 负责装配工具栏、设置菜单、日历主体和安排任务栏开关。
- `CalendarToolbar` 增加能力开关 props。完整日历页启用安排任务入口和周视图密度控制；嵌入式日历默认不显示这些控制，避免无效入口污染 `EmbeddedCalendarPanel`。
- `CalendarSurface` 继续选择月、周、列表视图，不承接业务状态。
- `WeekTimelineView` 承接周视图交互入口，但复杂计算下沉到 controller 纯函数。它通过 props 接收是否启用快速创建，完整日历页启用，嵌入式日历默认关闭。
- `useCalendarController` 暴露创建时间段任务、创建全天/跨天任务、调整时长和密度状态。
- `useTaskSchedulingActions` 继续处理已有排期 mutation，不负责创建任务。
- `SchedulingSidebar` 继续承接安排任务池展示，不直接调用排期 API。
- `calendarApi` 扩展 `createCalendarTask` 输入类型，允许时间段任务和跨天全天任务。组件不得直接绕过 calendar 模块调用 `tasksApi.createTask`。

新增或扩展的前端单元：

```txt
src/modules/calendar/controllers/weekTimelineInteraction.ts
src/modules/calendar/components/CalendarQuickCreatePopover.tsx
```

职责：

- `weekTimelineInteraction.ts`：纯函数计算点击/拖拽落点、15 分钟吸附、时间范围、日期范围、跨天全天创建输入。
- `CalendarSettings`：扩展 `weekTimelineDensity` 字段，定义三档密度和小时高度，继续使用现有 `CALENDAR_SETTINGS_STORAGE_KEY`。
- `CalendarQuickCreatePopover.tsx`：渲染创建浮层，提交前只管理标题、分类、错误展示和 loading 状态。

新增或扩展的契约：

```ts
type WeekTimelineDensity = 'compact' | 'standard' | 'comfortable';

type CalendarQuickCreateDraft =
  | {
      kind: 'timed';
      plannedDate: string;
      startAt: string;
      endAt: string;
      anchor: {x: number; y: number};
    }
  | {
      kind: 'all-day';
      plannedDate: string;
      plannedEndDate?: string;
      anchor: {x: number; y: number};
    };
```

`CalendarQuickCreateDraft` 由 `useCalendarController` 持有，`WeekTimelineView` 只负责请求打开 draft，`CalendarQuickCreatePopover` 只负责编辑和提交表单。

## 数据流

时间段快速创建：

1. `WeekTimelineView` 捕获空白时间轴点击或拖拽。
2. 通过纯函数计算 `{plannedDate, startAt, endAt, allDay:false}`，计算函数必须接收当前 `hourHeight`，不能假设 `1px = 1min`。
3. `WeekTimelineView` 请求 controller 打开创建浮层。
4. 用户输入标题和分类。
5. controller 调用扩展后的 `calendarApi.createCalendarTask`。
6. 成功后刷新日历数据、安排任务栏和上层任务数据。

跨天全天创建：

1. `WeekTimelineView` 捕获全天栏横向拖选。
2. 通过纯函数计算 `{plannedDate, plannedEndDate?, allDay:true}`。
3. 复用同一个创建浮层。
4. 保存后创建任务并刷新。

周视图展示跨天全天任务时，应使用连续条展示，而不是在每天重复显示同一个任务。可复用或扩展现有 `segmentAllDayTask` 逻辑，生成周视图全天栏分段。跨出当前周的任务要显示延续状态。

调整时长：

1. 用户拖动任务块底部 handle。
2. 纯函数根据拖动距离、`hourHeight` 和 15 分钟步进计算新 duration。
3. `resizeTimedTask` 调用已有排期接口更新 `endAt`。
4. 成功后刷新，失败则 toast。

### 事件优先级

周视图交互按以下优先级处理：

1. 外部拖拽排期 payload：当 `dragover/drop` 带有 calendar drag payload 时，只处理排期，不启动空白创建。
2. 时间段任务块拖拽：任务块自身 `dragstart` 不触发空白创建。
3. resize handle：`pointerdown` 后独占 resize 流程，并阻止冒泡到空白创建。
4. 空白快速创建：只在时间轴空白格或全天栏空白区域触发。

空白拖拽创建需要设置最小拖拽阈值。低于阈值按点击创建默认 60 分钟处理；超过阈值按拖拽范围创建。实现时需要用 pointer capture 或等价机制保证 pointerup 能正确收尾。

## API 和后端影响

本期不新增后端接口。

复用：

- `POST /api/tasks`
- `PATCH /api/tasks/:id/schedule`
- `GET /api/tasks`

现有后端排期规则已经支持：

- 时间段任务。
- 全天任务。
- 跨天全天任务。
- 拒绝跨天时间段任务。

前端必须遵守这些规则，不绕过 `scheduleRules`。

前端 API 边界：

- `calendarApi.createCalendarTask` 必须扩展为支持 `plannedEndDate/startAt/endAt/allDay` 的完整创建输入。
- `useCalendarController` 提供 `openQuickCreateDraft`、`closeQuickCreateDraft`、`submitQuickCreateDraft`。
- `WeekTimelineView` 和 `CalendarQuickCreatePopover` 不直接调用 `tasksApi`。

## 错误处理

- 没有分类：浮层内显示“请先创建分类”，禁止保存。
- 标题为空：浮层内显示“请输入任务标题”，不能提交。
- 创建失败：浮层内显示服务端错误，浮层保留，标题和分类输入不丢失。
- 调整时长失败：显示错误，刷新日历回到服务端状态。
- 快速创建选择跨天时间段：前端不产生该输入。
- 多选任务拖到时间轴：继续沿用现有拒绝逻辑。

## 验收标准

1. 用户可以在周视图时间轴空白处点击创建默认 60 分钟时间段任务。
2. 用户可以在周视图时间轴空白处向上或向下拖拽，创建 15 分钟步进的时间段任务。
3. 用户可以在周视图顶部全天栏拖选多个日期创建跨天全天任务。
4. 时间段快速创建和跨天全天创建展示一致的原地浮层交互：标题、分类、取消、保存和错误展示一致。
5. 取消浮层不会创建占位任务。
6. 保存失败时浮层输入不丢失。
7. 用户可以拖动时间段任务底部边缘调整任务时长。
8. 调整时长不能小于 15 分钟，不能跨出当天；接近日末时结束时间不晚于 `23:59`。
9. 周视图可以在紧凑、标准、宽松三档密度之间切换。
10. 密度选择通过现有 `CalendarSettings` 在前端本地保留。
11. 用户可以打开和关闭安排任务栏。
12. 关闭安排任务栏后完整日历页不再渲染 `SchedulingSidebar`，日历主区域使用单列布局。
13. 现有未安排任务拖拽排期继续可用。
14. 本期不出现没有真实模型支撑的标签或优先级 UI。
15. 嵌入式日历不显示安排任务入口，也不显示周视图密度控制。
16. 新建跨天全天任务后，周视图全天栏以连续条展示该任务，而不是每天重复一条。

## 风险

### 周视图组件继续变胖

`WeekTimelineView` 已经承接全天栏、时间轴、任务块和 resize。新增快速创建后，如果把计算逻辑直接写在组件里，会迅速失控。必须把时间计算、日期范围和密度规则抽成纯函数。

### 事件冲突

周视图已有拖拽排期和时间段任务移动。新增空白拖拽创建后，需要明确事件区域：任务块拖拽优先于空白创建，安排任务拖入优先于空白创建。

### 时间换算漂移风险

密度改变后，像素到分钟的换算不能散落在组件里。drop、quick create、resize 和任务块定位必须共享同一套 `hourHeight` 计算规则，否则视觉位置和提交时间会漂移。

### 假复刻风险

滴答截图包含标签和优先级，但 PlanTodo 当前没有对应模型。本期不得做假 tab。标签和优先级必须作为后续独立规格处理。

### 视觉伴随临时文件

设计过程中生成的 `.superpowers/` 目录是本地临时产物，不进入版本控制。
